Platform Data Flow & API Architecture

Request → Auth → API → Domain → Kafka/MQ → DB (schema-per-tenant)

① Request Flow

  • • Browser → Next.js Middleware (JWT validation)
  • • Middleware → Route Handler (Server Component)
  • • Route Handler → API Route (Server Action)
  • • API Route → Zod validation (@repo/shared-types)
  • • Validated → @repo/db (Prisma singleton)

② Authentication Flow

  • • User → /login → NextAuth GoogleProvider
  • • Google OAuth 2.0 + PKCE redirect
  • • Callback → Account + Session stored in tenant schema
  • • JWT session → Browser cookie (HttpOnly)
  • • Next.js Middleware re-validates on every request

③ Multi-Tenant Isolation

  • • Railway PostgreSQL (schema-per-tenant)
  • • Tenant identified via subdomain / JWT claim
  • • Prisma SET search_path = tenant_{name}
  • • All queries scoped to current tenant schema
  • • ADMIN role required for cross-tenant ops

④ Kafka Event Flow

  • • Domain events → kafka-topics.ts publishEvent()
  • • Topics: user.created, tenant.activated, billing.renewed
  • • Confluent Cloud (prod) / docker-compose (dev)
  • • KafkaJS Producer in API Route
  • • Dead Letter Topic (DLT) for failed messages

⑤ CI/CD Pipeline

  • • GitHub Actions: lint → type-check → build
  • • On main branch: prisma migrate deploy
  • • Vercel auto-deploy (preview per PR)
  • • Railway: managed PostgreSQL
  • • Ekko manual deploy for staging/prod

⑥ Shared Packages

  • • @repo/db: Prisma singleton + central schema
  • • @repo/auth: createAuthConfig() factory
  • • @repo/shared-types: Zod validators + domain events
  • • @repo/ui: shadcn/ui base components
  • • @repo/shared-config: ESLint + TS presets
